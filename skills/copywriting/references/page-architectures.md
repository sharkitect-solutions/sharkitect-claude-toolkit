# Page Architectures and Section-Ordering Logic

## Section Ordering Principles

The order of sections is NOT arbitrary. It follows the reader's psychological state as they scroll.

### The Persuasion Sequence

Every high-converting page follows this emotional arc:

1. **Hook** -- Match their mental state on arrival (awareness stage + traffic source)
2. **Credibility** -- Prove you are worth their time (social proof)
3. **Problem** -- Articulate their pain better than they can
4. **Solution** -- Show the bridge from pain to outcome
5. **Mechanism** -- Explain HOW (reduces perceived risk)
6. **Proof** -- Evidence the mechanism works (testimonials, case studies, data)
7. **Objections** -- Remove remaining friction
8. **Action** -- Clear, low-risk next step

You can skip sections or reorder slightly, but NEVER violate these rules:
- Hook always comes first
- Credibility must appear before you ask for anything
- Problem must come before solution (otherwise solution feels disconnected)
- Proof must come before the final CTA

### Section Selection by Traffic Source

| Traffic Source | What They Already Know | Emphasize | De-emphasize |
|---------------|----------------------|-----------|-------------|
| Paid search (branded) | Your product name, basic offering | Differentiators, pricing, social proof | Problem education |
| Paid search (non-branded) | The problem, solution category | Why you vs alternatives, mechanism | Problem articulation (they know it) |
| Social media ads | Almost nothing -- interrupted their feed | Hook, problem agitation, curiosity | Technical details, pricing |
| Organic search | The topic, some context from SERP | Depth, authority, comprehensive answer | Hard sell (they came for info) |
| Email | Your brand, prior relationship | What's new, specific offer, urgency | Brand introduction, trust-building |
| Referral | Someone vouched for you | Social proof, quick demo, easy start | Extensive credibility section |

## Page Architectures by Type

### Homepage Architecture

**Job**: Serve multiple audiences without being generic. Route visitors to the right path.

**Diagnostic Questions:**
- How many distinct buyer personas visit this page?
- What percentage of traffic is branded vs non-branded?
- Is the product simple (one use case) or complex (many)?

**Recommended Structure:**

```
1. Hero: Broadest value proposition that resonates with ALL segments
2. Social proof bar: Logos or aggregate stats
3. "Who it's for" routing section: 2-4 personas with paths
4. Top 3 benefits (shared across segments)
5. How it works: 3-step overview
6. Testimonials: One per primary segment
7. Integrations/partners (if applicable)
8. Final CTA: Lowest-friction entry point
```

**Homepage-Specific Rules:**
- The hero MUST work for your broadest audience -- do not niche down here
- Navigation stays visible (unlike landing pages)
- Provide escape routes for researchers (resources, about, docs)
- Keep the page scannable -- visitors decide in 5-10 seconds if they will stay

### Landing Page Architecture

**Job**: Single message, single conversion action, no distractions.

**Diagnostic Questions:**
- What awareness stage is the traffic in?
- What is the ONE action we want?
- What is the traffic source? (This determines the opening)

**Recommended Structure by Awareness Stage:**

**Problem-Unaware (longest page):**
```
1. Hook: Question or story that highlights an unrealized problem
2. Problem education: Data or narrative showing the problem is real
3. Cost of inaction: What happens if they don't solve it
4. Solution introduction: Your approach
5. How it works
6. Benefits with proof
7. Case study
8. Objections/FAQ
9. Guarantee/risk reversal
10. Final CTA
```

**Problem-Aware (medium page):**
```
1. Hook: "If you [symptom], you are not alone"
2. Problem agitation: Show you understand deeply
3. Solution: Bridge from problem to your product
4. Benefits with proof
5. How it works
6. Testimonials
7. Comparison to alternatives
8. FAQ
9. Final CTA with guarantee
```

**Solution-Aware (shorter page):**
```
1. Hook: Why THIS solution vs others
2. Key differentiators
3. Social proof (heavy -- they are comparing)
4. How it works
5. Pricing teaser or trial offer
6. FAQ focused on switching costs
7. Final CTA
```

**Most-Aware (shortest page):**
```
1. Offer headline (specific, time-sensitive if genuine)
2. Recap of key value
3. Social proof
4. Pricing/offer details
5. Guarantee
6. CTA
```

**Landing Page Rules:**
- Remove main navigation (reduce exit points)
- One CTA repeated 2-3 times on long pages
- Message-match the headline to the ad/email that sent them here
- Mobile: every section must be readable without horizontal scroll

### Pricing Page Architecture

**Job**: Help visitors choose the right plan and feel confident about it.

**Diagnostic Questions:**
- How many plans exist?
- Is there a free tier or trial?
- What is the biggest objection at the pricing stage?

**Recommended Structure:**
```
1. Headline: Frame the value, not the cost ("Plans that grow with you")
2. Plan comparison table: Feature matrix with clear recommendation
3. Most popular plan highlighted visually
4. FAQ: Billing, cancellation, upgrades
5. Social proof: Customer who upgraded and saw results
6. Guarantee/risk reversal
7. "Not sure which plan?" CTA to talk to sales
```

**Pricing Page Rules:**
- ALWAYS recommend a plan visually (highlight, "Most Popular" badge)
- Put annual pricing first if you offer a discount
- Address "what happens if I outgrow this plan?" -- show the upgrade path
- Do not hide pricing behind "Contact Us" unless truly enterprise-only

### Feature Page Architecture

**Job**: Connect feature to benefit to outcome. Differentiate from competitors.

**Recommended Structure:**
```
1. Hero: Feature name + benefit headline (not just the feature name)
2. Visual: Screenshot or demo of the feature in action
3. Problem this feature solves
4. How the feature works (walkthrough)
5. Use cases: 2-3 scenarios with different personas
6. Comparison: How this is different from competitor's version
7. Testimonial from user who loves this feature
8. CTA: Try this feature (link to relevant trial path)
```

### About Page Architecture

**Job**: Build trust and emotional connection. Still a marketing page.

**Recommended Structure:**
```
1. Origin story: Why you built this (founder motivation)
2. Mission/values: What you believe (connect to customer benefit)
3. Team section: Real people, real photos
4. Traction/milestones: Company growth, customer count
5. Culture: How you work (builds employer brand too)
6. CTA: Still include one -- "Join us" or "See what we built"
```

## Section Types Reference

### Engagement Sections (Break up feature/benefit monotony)

| Section Type | Purpose | When to Use |
|-------------|---------|-------------|
| How It Works (numbered steps) | Reduce complexity perception | Always on landing pages |
| Competitor comparison | Differentiate | Solution-aware traffic |
| Founder story/manifesto | Emotional connection | About pages, brand-driven landing pages |
| Use cases / personas | Self-identification | Complex products with multiple audiences |
| Interactive demo / product tour | Show, don't tell | High-consideration products |
| Integrations / partners | Stack compatibility | B2B SaaS, developer tools |
| Pricing teaser | Move decision forward | Non-pricing pages where price is an objection |

### Proof Sections (Build credibility)

| Section Type | Strength | When to Use |
|-------------|----------|-------------|
| Customer logos | Quick authority | Above fold, early on page |
| Aggregate stats ("10,000+ teams") | Scale proof | Near hero, social proof bar |
| Star ratings with count | Third-party validation | E-commerce, SaaS with review presence |
| Full testimonials | Deep credibility | Mid-page, after benefits |
| Case study snippets | Result-oriented proof | Late-page, before final CTA |
| Video testimonials | Highest trust | Key conversion pages |

### Friction-Reduction Sections

| Section Type | Objection Addressed | When to Use |
|-------------|-------------------|-------------|
| FAQ | Multiple objections | Every page, near bottom |
| Guarantee / risk reversal | "What if it doesn't work?" | Before final CTA |
| "Built for [audience]" | "Is this for me?" | When audience self-selection matters |
| Security/compliance badges | "Is my data safe?" | Enterprise, fintech, health |
| Free trial terms | "Will I get locked in?" | SaaS pricing, final CTA area |

## Anti-Pattern: The Feature Stack

**Bad page structure (common):**
```
Hero -> Feature 1 -> Feature 2 -> Feature 3 -> Feature 4 -> Feature 5 -> CTA
```

This fails because:
- No emotional arc -- just a list
- No proof that any feature delivers results
- No problem context -- features without problems are meaningless
- No variety -- reader fatigue sets in by Feature 3
- No objection handling -- visitor leaves with unresolved doubts

**Fix:** Interleave features with proof, story, and friction-reduction sections.
