# Plan: Upgrade Fantastic Floors Error Logger

## Context

The Fantastic Floors Global Error Handler (`oQDSXQdTyQgEFIyVMIwFx`) classifies errors as `system`, `user_data`, or `user_data_unknown` via the Classify Error code node. The ERROR Logger currently fires in parallel with the Is User Error? check — meaning it logs ALL errors identically, ignoring classification data (error type, fix instructions, category).

**Goal:** Split logging into two branch-specific loggers so user errors and system errors are logged differently, with auto-populated notes and clear status labels.

### Current Workflow Topology

```
Error Trigger → Normalize Error Payload → Classify Error
  → [parallel] → Is User Error? → (true)  → Fetch Failed Execution → ... → Send Monday Notification
  │                              → (false) → ERROR Message - Slack (to Chris)
  │                                        → ERROR Message - GMail [DISABLED]
  └──────────→ ERROR Logger (Airtable — logs ALL errors identically)
```

**Problem:** ERROR Logger fires from Classify Error before the branch split. It logs every error the same way — no Status, no Notes, no distinction between user and system.

### Data Available from Classify Error

- `errorType`: `"system"`, `"user_data"`, or `"user_data_unknown"`
- `userAction`: Fix instructions text (user errors) or empty string (system)
- `errorCategory`: Pattern name like `"missing_qbo_item"`, `"multiple_product_service"`, etc.

## Implementation Plan

### New Architecture

Move logging from before the branch split to AFTER it — one logger per branch. On the system branch, Slack fires AFTER the LLM so Chris gets the diagnostic hint in the same notification:

```
Classify Error → Is User Error?
  → (true)  → [parallel] → Fetch Failed Execution → ... (existing user flow)
  │                       → User ERROR Logger (Status="User", Notes=userAction)
  │
  → (false) → Generate System Notes (LLM) → [parallel] → System ERROR Logger (Airtable)
                                                         → ERROR Message - Slack (enhanced with LLM hint)
```

**Why this architecture:**
- Clean separation — each branch handles its own logging with the right data
- LLM only fires for system errors (no wasted calls on user errors)
- Slack gets the LLM diagnostic hint — Chris can start fixing immediately from the notification without checking Airtable
- Airtable serves as permanent documentation backup (Slack messages can be deleted to keep channels clean)
- Existing user flow completely untouched — logger attaches in parallel

### Step 2: Repurpose Existing ERROR Logger → User ERROR Logger

The existing ERROR Logger node (`6aa2492c-...`) already has the correct Airtable credential and 9 field mappings. We'll:

1. **Rename** it to "User ERROR Logger"
2. **Add 2 new field mappings:**
   - **Status**: `={{ $json.errorType === 'user_data_unknown' ? 'User (Unknown)' : 'User' }}`
   - **Notes**: `={{ $json.userAction || '' }}` (fix instructions from Classify Error)
3. **Disconnect** from Classify Error output
4. **Connect** to Is User Error? true output (in parallel with Fetch Failed Execution)

No new node needed — just repurpose + rewire the existing one.

### Step 3: Add "Generate System Notes" Code Node (LLM)

New Code node on the system branch. Calls Haiku for a diagnostic hint.

```javascript
// Generate diagnostic hint for system errors via Haiku
const errorMessage = $json.errorMessage || '';
const errorDetails = $json.errorDetails || '';
const failedNode = $json.failedNode || '';
const workflowName = $json.workflowName || '';

try {
  const response = await $helpers.httpRequest({
    method: 'POST',
    url: 'https://api.anthropic.com/v1/messages',
    headers: {
      'x-api-key': '<ANTHROPIC_API_KEY>',
      'anthropic-version': '2023-06-01',
      'content-type': 'application/json'
    },
    body: JSON.stringify({
      model: 'claude-haiku-4-5-20251001',
      max_tokens: 200,
      messages: [{
        role: 'user',
        content: `You are an n8n workflow error diagnostician for a flooring company's automation system. Given this system error, provide a 1-3 sentence actionable diagnostic hint for the engineer.

Error: ${errorMessage}
Details: ${errorDetails.slice(0, 500)}
Failed Node: ${failedNode}
Workflow: ${workflowName}

Respond with ONLY the diagnostic hint. No preamble.`
      }]
    }),
    json: true
  });

  const notes = response?.content?.[0]?.text || '';
  return [{ json: { ...$json, notes } }];
} catch (e) {
  // LLM failure must not block logging
  return [{ json: { ...$json, notes: `[LLM unavailable] ${e.message || ''}` } }];
}
```

**API key handling:** Determine during implementation — check if n8n instance has Anthropic key as env var or credential. Fallback: inline in code.

**Cost:** ~$0.001 per system error. Only fires on system errors.

**Position:** Connected from Is User Error? false output (sequential, not parallel — Slack and Logger both come after).

### Step 5: Rewire ERROR Message - Slack (Enhanced)

The existing Slack node currently fires from Is User Error? false. We'll:

1. **Disconnect** from Is User Error? false output
2. **Connect** from Generate System Notes output (in parallel with System ERROR Logger)
3. **Update the message** to include the LLM diagnostic hint from `$json.notes`

Enhanced Slack message format:
```

*Workflow:* {{ $json.workflowName }}
*Failed Node:* {{ $json.failedNode }}
*Error:* {{ $json.errorMessage }}
*Execution:* {{ $json.executionUrl }}

:brain: *Diagnostic Hint:*
{{ $json.notes }}
```

This way Chris gets the error details AND the LLM's suggested fix in one Slack message — no need to open Airtable.
